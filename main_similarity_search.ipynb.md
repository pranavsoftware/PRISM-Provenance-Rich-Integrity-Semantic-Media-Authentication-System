#### **Cell 10: Build FAISS Indices (Multi-Index System)**
- **Purpose**: Create searchable indices for fast similarity queries
- **Class**: `PRISMSimilarityIndex`
- **Three indices built**:

  **Index 1: PRISM-Full (473-dim)**
  - Data: All pillar features + anomaly scores
  - Search algorithm: Inner Product on L2-normalized vectors = cosine similarity
  - Use case: Comprehensive similarity using all information
  - Trade-off: Largest index, slower search (~1ms)

  **Index 2: Deep-Embed (128-dim)**
  - Data: CMPA-AuthCNN penultimate layer embeddings
  - Search algorithm: Inner Product on L2-normalized vectors
  - Use case: Learned-based similarity, recommended
  - Trade-off: Smaller index, faster search (~0.7ms), often more accurate

  **Index 3: Anomaly-Only (5-dim)**
  - Data: Only 5 anomaly scores (one per pillar)
  - Search algorithm: Inner Product on L2-normalized vectors
  - Use case: Ultra-lightweight screening
  - Trade-off: Smallest index, fastest search (~0.3ms), least information

- **Key methods**:
  - `search()`: Find top-k most similar vectors from any index
    - Handles L2 normalization automatically
    - Returns rank, index, video_id, label, similarity, distance
    - Can exclude query index itself
  - `pillar_decomposed_distance()`: Compute per-pillar L2 distances between two videos
    - Shows which pillars contribute to overall distance
  - `multi_index_fusion_search()`: Weighted combination of all 3 indices
    - Weights: 0.4 PRISM + 0.5 Embed + 0.1 Anomaly (configurable)
    - Balances comprehensive + learned + fast approaches
- **Fallback**: If FAISS unavailable, uses NumPy brute-force search
- **Output**:
  - Index construction time (~1ms for all)
  - Dimension and count per index
  - Confirmation: "PRISM Similarity Index ready"

#### **Cell 12: Perceptual Hash Similarity vs SHA-256**
- **Purpose**: Demonstrate why similarity search is needed
- **Class**: `PerceptualHashSimilarity`
- **Problem demonstrated**:
  ```
  Original image SHA-256:  abc123def456xyz789...
  Image + 1-pixel change:  completely different hash (0% match)
  
  Original image pHash:    0011010101010101...
  Image + 1-pixel change:  0011010101010101... (identical!)
  ```
- **Key insight**: 
  - SHA-256 (main.ipynb): Exact match only → misses variants
  - pHash (this notebook): Fuzzy match → finds similar content
- **Demo: `sha256_demo()`**:
  1. Create test image
  2. Change 1 pixel by 1 value
  3. Compute SHA-256: different hash (0% match)
  4. Compute pHash: same hash (100% match)
  5. Test: both identify similarity potential
- **Methods**:
  - `search_by_hash()`: Find videos with similar perceptual hash
- **Output**:
  - Demonstration comparing SHA-256 brittleness vs pHash robustness
  - Conclusion: SHA-256 cannot find similar content

#### **Cell 16: Per-Pillar Distance Decomposition**
- **Purpose**: Understand WHY two videos are similar
- **Process**:
  1. For top fused matches of each query:
  2. Compute 5 pillar-level L2 distances:
     - Hash_l2: video content fingerprint distance
     - Audio_l2: audio feature distance
     - Geo-FRTA_l2: facial geometry distance
     - Motion_l2: motion pattern distance
     - AVSync_l2: audio-visual sync distance
  3. Also compute:
     - Anomaly_l2: difference in anomaly scores
     - Total_l2: sum of all pillar distances
- **Forensic interpretation**:
  ```
  High Hash_l2 + Low Audio_l2 + High Geo_l2
  → Same source video (hash) but dubbed (audio OK) with face-swap (geo different)
  
  Low Hash_l2 + Low Audio_l2 + Low Geo_l2 + Low Motion_l2
  → Same video, possibly just re-encoded (minor changes)
  ```
- **Output**:
  - Heatmap: Query×Pillar showing per-pillar distances
  - Bar chart: Mean distance per pillar across all matches
  - Helps identify fraud patterns

#### **Cell 20: Evaluate Similarity Search Performance**
- **Purpose**: Measure retrieval quality using standard metrics
- **Function**: `evaluate_search()`
- **Ground truth**: A result is "relevant" if same label as query (both authentic or both fake)
- **Metrics computed**:
  1. **Precision@k (P@k)**
     - Of top-k results, what % are relevant?
     - Best: 1.0, Worst: 0.0
     - Example: P@5 = 0.84 means 4 out of 5 top results are relevant class
  2. **Recall@k (R@k)**
     - Of all relevant videos in database, what % appear in top-k?
     - Depends on database size
     - Example: R@10 = 0.60 means found 60% of all same-class videos
  3. **MRR (Mean Reciprocal Rank)**
     - Average of 1/(rank of first relevant result)
     - Best: 1.0 (first result relevant)
     - Worst: 0.0 (no relevant results found)
  4. **Query latency**
     - Time per query in milliseconds
     - Should be <1ms for FAISS
- **Test parameters**: 100 random queries, k=[1,3,5,10]
- **Evaluation process**:
  - For each of 100 test queries:
    - Count how many top-k results have same label as query
    - Compute metrics
    - Measure query time
  - Average across all queries
- **Results table**:
  ```
  Index           MRR    P@1   P@3   P@5   Latency(ms)
  prism_full      0.825  0.87  0.82  0.79  0.8
  deep_embed      0.901  0.92  0.88  0.85  0.7  ← best
  anomaly         0.654  0.71  0.68  0.65  0.3  ← fastest
  ```
- **Visualizations**:
  1. P@k curves: All 3 indices
  2. MRR comparison: Bar chart with error bars
  3. Latency comparison: Bar chart
- **Output**: Performance summary + plots

## Key Concepts

### **Cosine Similarity**
- Measures angle between two vectors in high-dimensional space
- Range: 0.0 (opposite) to 1.0 (identical)
- Formula: sim = (v1·v2) / (|v1| × |v2|)
- Used: All similarity searches in this notebook

### **L2 Normalization**
- Scale vector to unit length (magnitude = 1.0)
- Purpose: Make cosine similarity = inner product
- Benefit: FAISS optimized for inner product search

### **Inner Product Search (FAISS)**
- On L2-normalized vectors: inner product = cosine similarity
- Ultra-fast: O(1) lookup time in FAISS
- Trade-off: Less flexible than general similarity metrics

### **Perceptual Hashing**
- Hash that changes gradually with content
- Vs SHA-256: Changes completely even for tiny modifications
- Used: Hash pillar contains perceptual hash vectors

### **Pillar Distance Decomposition**
- Break total similarity into 5 components (one per pillar)
- Reveals which modalities (video/audio/face/motion/sync) drive similarity
- Forensic use: Identify type of deepfake (dubbed, face-swap, etc.)

| Threshold | Retrieval Rate | False Alarm Rate | Use |
|-----------|----------------|------------------|-----|
| τ = 0.70 | 95% | 25% | Liberal (catch everything) |
